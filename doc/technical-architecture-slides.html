SilverRetort 技术架构

面向智能体沙盒的通用前端

桌面端拉起本地 Web UI 与 Python API，通过 Hermes / Switch / MCP Bridge 连接本地或远程容器，让智能体在受控 workspace 中执行，并把结果以 artifact 组件回到前端。

序章

核心特性

Sandbox

沙盒执行，skill 自进化

Hermes 运行在独立 home / workspace 中；远程模式由 Docker 卷持久化，skill、脚本和领域知识留在沙盒侧迭代。

apps/hermes/main.py · apps/hermes/workspaces.py

Local MCP

连接用户本地程序

用户配置的 MCP 仅允许 loopback URL，经 uvicorn bridge 反向转发到远端 Hermes，可调用本地专用软件或设备服务。

bridge_client.py · local_mcp_servers.py

Workflow

企业工作流接入

Hermes gateway 提供 OpenAI-compatible API，SilverRetort 侧保留模型、slash command、workspace 和 MCP 边界，适合接 Dify 等流程编排。

hermes_client.py · relay.py

第一章

项目定位：通用智能体沙盒前端

SilverRetort 不是只绑定一个 bot 的聊天壳，而是把“会话、工作区、沙盒连接、模型控制、artifact 渲染器、本地 MCP”抽成前端平台能力。

  • 一个 UI 可管理多个 workspace，每个 workspace 绑定一个 switch profile。
  • 当前主要接 Hermes；后续可通过标准 OpenAI / Responses 兼容接口接 PI 等框架。
  • 垂直领域能力以 skill、artifact renderer、MCP 工具三种形态沉淀。
packages/protocol/types.ts: Workspace / SwitchProfile / Artifact / ChatEvent 是前后端共享边界。

路线图

从 Hermes 扩展到更多智能体框架

  • 短期：强化 Hermes 容器、workspacePort、artifact bridge 和本地 MCP 的稳定性。
  • 中期：沉淀领域 skill 包和专用 artifact renderer，例如机器人、EDA、仿真与企业审批流。
  • 长期：通过 OpenAI-compatible adapter 接入 PI 等框架，同时保持 UI、workspace、MCP、artifact 协议不变。

核心判断：前端平台不直接绑定某个 agent runtime，而是稳定提供“沙盒连接 + 工作区 + 本地能力桥 + 交互组件”四个横向能力。
